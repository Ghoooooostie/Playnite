# Switch 智能导入插件设计

## 目标

做一个 Playnite 插件，专门解决 Switch 游戏导入时把本体、补丁多个版本、DLC 全部当成独立游戏导入的问题。

第一版只支持 Switch，只提供主菜单入口，不改 Playnite 原生导入窗口。

## 范围

本次只做下面这些能力：

- 主菜单入口打开“Switch 智能导入”
- 支持配置多个扫描目录
- 支持手动扫描
- 支持定期扫描
- 定期扫描结果只进入待确认列表，不直接入库
- 识别 Switch 本体、补丁、DLC
- 同一游戏只保留本体作为候选导入项
- 多个补丁版本只保留最高版本作为附加信息
- DLC 默认不进入待确认列表
- 支持规则配置和忽略配置

本次不做下面这些能力：

- 不替换 Playnite 原生模拟器导入窗口
- 不做全模拟器通用规则
- 不自动直接导入数据库
- 不做网络元数据抓取

## 用户场景

用户的目录结构并不统一，第一版不能把规则绑死在“本体/补丁/DLC 分目录”上。

常见情况至少有两类：

- 分目录存放：本体、补丁、DLC 分开，补丁可能再按版本分子目录存放
- 同目录混放：本体和补丁文件直接放在同一个目录，依赖文件名里的 `[Base]`、`[Update]`、`[DLC]` 标记区分

插件需要避免出现下面这种错误结果：

- 本体被导入 1 次
- 补丁 `v1.0.1`、`v1.0.2`、`v1.0.3` 各被导入 1 次
- DLC 也被当成独立游戏导入

正确结果是：

- 同一游戏只出现 1 条待导入记录
- 补丁只作为该游戏的版本信息存在
- DLC 默认跳过

## 总体方案

插件使用独立扫描流程，不复用 Playnite 原生“模拟器游戏导入”界面。

插件从主菜单打开一个“智能导入中心”，由它完成下面三件事：

1. 读取配置中的多个扫描目录
2. 扫描并识别 Switch 本体、补丁、DLC
3. 生成待确认列表，等待用户手动确认导入

## 插件入口

第一版只放在主菜单。

入口名称：

- `Switch 智能导入`

主菜单下提供三个动作：

- 打开智能导入中心
- 立即扫描
- 打开插件设置

这样用户既能完整进入管理页面，也能直接触发一次扫描。

## 设置项

插件设置页提供以下配置。

### 扫描目录

支持多个目录，每个目录单独保存。

每条目录配置包含：

- 显示名称
- 路径
- 是否启用
- 优先级
- 类型提示：本体 / 补丁 / DLC / 自动判断

目录类型提示只用于增强判断，不作为唯一依据。

### 扫描策略

- 是否开启定期扫描
- 扫描间隔（分钟）
- 启动 Playnite 后是否自动做一次扫描
- 扫描时是否包含子目录

### 过滤策略

- DLC 默认跳过
- 是否在待确认列表中显示 DLC
- 是否记录补丁最高版本信息
- 是否忽略版本号差异导致的重复项
- 自定义忽略关键词
- 自定义忽略目录

### 导入策略

- 默认导入使用的模拟器
- 默认导入使用的模拟器配置
- 默认平台
- 是否使用相对路径导入
- 元数据来源

### 元数据来源

第一版提供一个可选的“导入后自动补资料”设置。

当前只需要支持：

- 不选择
- `Switch Local Metadata`

行为规则：

- 如果元数据来源未填写，则导入后不自动处理任何资料
- 如果元数据来源填写为 `Switch Local Metadata`，则导入完成后自动对这批新游戏执行一次资料刷新
- 这次资料刷新是全量刷新，不是只补缺失字段

全量刷新范围按 `Switch Local Metadata` 当前支持字段执行：

- 名称
- 开发商
- 发行商
- 平台
- 链接
- 图标
- 封面
- 安装大小

如果 `Switch Local Metadata` 插件未安装、被禁用，或者它自己的路径配置不完整，则本插件不兜底猜测、不改为部分刷新，直接提示用户并跳过自动补资料。

## 扫描与识别规则

第一版按“包名标记 + Title ID + 文件名 + 目录结构”做严格识别，不做模糊猜测兜底。

支持的候选文件扩展名：

- `.nsp`
- `.xci`
- `.nsz`
- `.xcz`

### 识别优先级

识别顺序固定如下：

1. 文件名中的包类型标记
2. 文件名中的 Title ID
3. 文件名中的版本信息
4. 所在目录名称和层级

也就是：

- `[Base]`、`[Update]`、`[DLC]` 这类包名标记优先级最高
- 目录是不是叫“本体”“补丁”“DLC”只能作为辅助，不是主判断条件

### 文件名规则

第一版重点支持你现在这种常见命名：

- `游戏名 [TitleId][来源][版本][Base].nsp`
- `游戏名 [TitleId][来源][版本][Update].nsp`
- `游戏名 [TitleId][来源][版本][DLC].nsp`

从文件名里优先提取：

- 游戏显示名
- Title ID
- 包类型：Base / Update / DLC
- 版本号

如果文件名能明确提取出这些信息，就直接按文件名结果判定，不再依赖目录结构。

### 本体识别

满足下面任一条件时，优先认定为本体候选：

- 文件名包含 `[Base]`
- 文件位于标记为“本体”的扫描目录
- 文件位于名称包含“本体”“base”“game”这类主内容目录
- 文件名不包含补丁和 DLC 关键词，且同目录结构表现为主游戏文件

### 补丁识别

满足下面任一条件时，认定为补丁：

- 文件名包含 `[Update]`
- 文件位于标记为“补丁”的扫描目录
- 父目录或祖先目录包含“补丁”“update”“patch”
- 路径中存在明显版本目录，例如 `1.0.1`、`1.0.2`
- 文件名包含补丁关键词或版本升级信息

同一游戏存在多个补丁时：

- 只保留最高版本补丁
- 低版本补丁只记为被过滤原因，不进入待确认列表

### DLC 识别

满足下面任一条件时，认定为 DLC：

- 文件名包含 `[DLC]`
- 文件位于标记为“DLC”的扫描目录
- 父目录或祖先目录包含“dlc”“追加内容”“附加内容”
- 文件名包含 DLC 关键词

第一版默认直接跳过 DLC。

### 去重规则

去重不按“完整文件名”做，而按“规范化后的游戏主名 + 关联 Title ID”做。

规范化时去掉：

- 版本号，如 `v1.0.1`
- 补丁标记，如 `update`、`patch`
- DLC 标记
- 常见发布后缀
- 多余平台后缀，如 `for Nintendo Switch`

如果文件名里带 Title ID，则优先用它帮助归并同一游戏：

- Base 和 Update 的 Title ID 不完全相同时，也要允许按“同名 + Base/Update 关系”归组
- 同名不同游戏不能只靠名字硬合并，优先参考 Title ID 和包类型

第一版的目标不是做通用破解包数据库，而是先稳定处理你现在最常见的这两种情况：

- 同目录下同时存在 Base 和 Update
- 分目录下存在多个 Update 版本

去重后的结果按下面规则落地：

- 有本体时，只保留本体
- 没找到本体但只找到补丁时，不自动导入，进待确认列表并标记“缺少本体”
- 同名多个本体时，按路径优先级和文件更新时间选 1 个主候选，其他记为重复项

## 待确认列表

扫描完成后生成待确认列表，不直接入库。

每条记录显示：

- 游戏名
- 识别类型
- 本体路径
- 最高补丁版本
- 过滤结果摘要
- 默认是否勾选导入

同时显示被跳过项和原因，例如：

- 跳过 DLC
- 跳过低版本补丁
- 跳过重复本体
- 只发现补丁未发现本体

用户可在待确认列表中：

- 勾选/取消勾选
- 手动改名
- 手动改平台
- 手动忽略某条记录
- 把某条规则加入忽略关键词或忽略目录

## 导入流程

用户在待确认列表中点击导入后，插件将选中的候选项转换成 Playnite 游戏并写入数据库。

导入时：

- 每个候选游戏只生成 1 个 Playnite 游戏
- 游戏 ROM 路径指向最终保留的本体文件
- 补丁和 DLC 不作为额外游戏导入
- 最高补丁版本写入游戏版本信息或插件附加数据
- 如果配置了元数据来源，则导入完成后立即对新导入游戏批量执行一次资料刷新

如果候选项缺少本体，则默认不允许导入，除非用户后续版本明确要求支持这种例外。

### 自动补资料

如果用户在设置中选择了 `Switch Local Metadata`，导入后流程增加一步：

1. 收集本次成功导入的新游戏
2. 查找已加载的 `Switch Local Metadata` 元数据插件
3. 只使用这个来源，对每个游戏执行一次全量资料刷新
4. 把结果写回 Playnite 数据库

这里的“全量刷新”含义是：

- 不看字段是否已有值
- 只要来源返回了值，就覆盖到游戏上

这里不会自动调用别的元数据来源，也不会混合多个来源。

## 定期扫描

定期扫描由插件内部定时器驱动。

行为规则：

- 到点后自动扫描启用的目录
- 扫描结果只更新待确认列表
- 不自动写入 Playnite 数据库
- 若有新增候选项，给出通知提示

如果 Playnite 当前正在做别的导入或关闭过程中，本次扫描直接跳过，不做并发兜底补救。

## 数据保存

插件保存以下数据：

- 插件设置
- 扫描目录列表
- 自定义过滤规则
- 待确认列表缓存
- 上次扫描时间
- 已忽略项目记录

待确认列表缓存用于：

- Playnite 重启后继续确认
- 定期扫描追加新候选项

## 界面结构

第一版界面保持简单，分成两个页面。

### 设置页

内容：

- 扫描目录管理
- 扫描间隔设置
- 过滤规则设置
- 默认导入设置

### 智能导入中心

内容：

- 扫描按钮
- 最近扫描时间
- 待确认列表
- 已跳过列表
- 导入按钮

## 错误处理

插件遇到异常时直接暴露明确结果，不做静默吞错。

需要明确提示的情况：

- 目录不存在
- 目录无权限
- 模拟器未配置
- 默认导入配置不完整
- 扫描结果只有补丁没有本体
- 导入入库失败
- 用户选择了 `Switch Local Metadata`，但插件未安装或未启用
- 用户选择了 `Switch Local Metadata`，但它自己的工具路径配置不完整
- 自动补资料执行失败

## 测试范围

第一版至少覆盖下面这些测试。

### 识别测试

- 本体目录中单个本体文件
- 同游戏多个补丁版本目录
- DLC 目录
- 路径和文件名同时带版本号
- 日文游戏名 + 英文后缀
- 同目录下同时存在 `[Base]` 和 `[Update]`
- 仅靠文件名标记即可识别 Base / Update / DLC

### 去重测试

- 同一本体重复出现
- 本体和补丁同时出现
- 只有补丁没有本体
- 多个目录都扫到同一个游戏
- 同目录混放时只保留 Base，不把 Update 当独立游戏

### 流程测试

- 多目录扫描生成待确认列表
- 定期扫描只更新待确认列表
- 用户确认后导入成功
- 忽略规则保存后下次扫描生效
- 元数据来源为空时导入后不触发自动补资料
- 元数据来源为 `Switch Local Metadata` 时导入后触发全量刷新
- `Switch Local Metadata` 不可用时跳过并提示

## 实现拆分

建议拆成下面几个模块：

- `SwitchSmartImportPlugin`：插件入口和主菜单
- `SwitchSmartImportSettings`：设置读写
- `SwitchImportScanner`：目录扫描
- `SwitchPackageClassifier`：本体 / 补丁 / DLC 识别
- `SwitchCandidateMerger`：同游戏候选合并和去重
- `SwitchPendingImportStore`：待确认列表持久化
- `SwitchImportViewModel`：导入中心逻辑
- `SwitchImportSettingsViewModel`：设置页逻辑
- `SwitchImportExecutor`：把候选项写入 Playnite
- `SwitchMetadataRefreshService`：导入后元数据全量刷新

## 决策

第一版故意只做 Switch，因为当前用户痛点就是 Switch 文件结构导致的错误导入。

先做 Switch 专用，可以把“本体、补丁、DLC、多版本补丁”这几个规则做扎实，避免一开始做成泛化规则后误判更多。
